Chapter 4 · §2

Gaussian elimination

The previous section set the stage: a system of equations is an intersection of lines or hyperplanes, and three innocent manipulations let you pull it in any direction without changing its solutions. This section builds the mechanical recipe that solves any system in a finite number of steps — and which, as a bonus, powers three other major computations in this chapter. One algorithm, four uses.

The core idea is almost too simple to be true. Instead of inventing a new trick for each system, we fix a target form in advance — a presentation so clean the solution can be read off by eye — and apply the elementary operations in a disciplined order. The target form is called the reduced row-echelon form, abbreviated RREF.

What we're aiming for: the RREF

Before mapping out a route, you need to know where you're going. Take a look at these two versions of the same matrix. The one on the left is arbitrary, unreadable. The one on the right describes exactly the same system, but in a way that hands you the answer with no calculation.

Starting matrix After reduction (RREF) [ 2 4 -2 | 7 ] [ 1 0 0 | 3 ] [ 4 9 -3 | 13 ] ====> [ 0 1 0 | 2 ] [ -2 -3 7 | 1 ] [ 0 0 1 | 1 ]

If the last column holds the right-hand sides, the reduced version reads off directly as $x = 3$, $y = 2$, $z = 1$. All the algorithm's energy goes into turning the left-hand matrix into the right-hand one without ever losing a solution. Those $1$s stepping down the diagonal have a name: they are the pivots. Everything else is zero, and that austerity is what makes the matrix immediately readable.

The four formal conditions

A matrix is in RREF if, and only if, it satisfies the following four conditions.

  1. Pivots equal to one. In every nonzero row, the first nonzero entry equals exactly $1$. That's the pivot of that row.
  2. Descending staircase. Each pivot is strictly to the right of the pivot in the row above. The pivots form a diagonal that may skip columns but never moves backward.
  3. Pivot columns cleared. In every column that contains a pivot, that pivot is the only nonzero entry. Zeros above, zeros below.
  4. Zero rows at the bottom. Any row that is entirely zero, if there is one, sits all the way at the bottom.

Watch out for one detail: these conditions only constrain the columns that contain a pivot. Columns without a pivot — which we'll later call free columns — can hold any values. The matrix is genuinely in RREF even if a pivot-less column shows strange numbers. We'll come back at the end to interpret these stray columns.

The three operations we're allowed to perform

The previous section already introduced the three elementary row operations. We restate them here in a table, with their inverse operation, because it is reversibility that makes them legitimate. As long as we can always step back, no solution can be created or destroyed along the way.

# Operation Inverse
1 Swap two rows Swap them back
2 Multiply a row by k (k nonzero) Divide that row by k
3 Add k times one row to another Subtract k times that same row

Operation 1 is used to bring a row with a good pivot up to the right position. Operation 2 is used to normalize a pivot — that is, scale it down to $1$. Operation 3 is the real engine: it's the one that creates zeros where we want them, by combining two rows. The whole trick lies in the choice of coefficient.

The key formula: "what you want to kill, divided by the pivot"

Here's the central move. At some point during the algorithm, we have a row containing a pivot — call it $L_i$ — and we want to make the corresponding entry vanish in another row $L_j$, in the same column. We want to replace that entry of $L_j$ with zero without breaking the other entries. The technique is operation number 3, and the coefficient comes out without hesitation:

$$k = \frac{\text{entry of } L_j \text{ in the pivot column}}{\text{pivot value in } L_i}, \quad \text{then } L_j \leftarrow L_j - k \cdot L_i$$

Plainly: $k$ equals what we want to cancel, divided by the pivot. Why? Because if we subtract $k$ times $L_i$ from $L_j$, we subtract $k \cdot (\text{pivot})$ in that column — exactly what we wanted to cancel. The entry evaporates. The other columns of $L_j$ move too, but they'll be sorted out in turn when we work on their own column.

Now the simplification that makes the rest of the work pleasant. If we've normalized $L_i$ first — divided the whole row so that its pivot equals $1$ — then the division in the formula for $k$ vanishes:

$$k = \text{entry of } L_j \text{ in the pivot column} \quad (\text{because pivot} = 1)$$

No more division: $k$ is just the value we want to kill. That's why we normalize before eliminating. The order of substeps isn't decorative — it saves computation and cuts down on errors.

One engine, four uses. The same procedure that solves a system also computes the rank, the determinant, and the inverse of a matrix. That shared machinery is what gives chapter 4 its elegance.

REF or RREF: one step apart

In the literature, you'll find two echelon forms that are closely related but distinct. The REF (Row Echelon Form, without the "Reduced") only requires that the pivots trace a descending staircase and that there be zeros below the pivots; the pivots themselves can take any nonzero value, and above them the matrix can hold anything.

The RREF demands all of that plus two more things: the pivots equal exactly $1$, and the zeros also extend above the pivots. The REF is intermediate, the RREF is terminal. For immediate readability, RREF is what we want. The algorithmic difference is minor: same formula for $k$, but we eliminate in all other rows (above and below), and we normalize each pivot to one along the way.

The structure of the algorithm

Now we can describe the machinery. The algorithm sweeps the matrix column by column from left to right. For each column, it tries to place a pivot. If it succeeds, it cleans up that whole column; if not, it moves on to the next one.

pivot_row ← 0 for each column c, from first to last: if pivot_row has gone past the last row: stop (no rows left) look in column c, among rows >= pivot_row, for a row whose entry is not zero. if none qualifies: free column, move to the next column (pivot_row stays at its current value) otherwise, calling j the row found: 1. swap rows j and pivot_row (if they differ) 2. normalize pivot_row: divide the whole row by the pivot 3. eliminate column c in all other rows pivot_row ← pivot_row + 1

Two indices wander through this loop, and it is crucial that they do not advance at the same rate. The index c walks through the columns one by one, from start to finish. The index pivot_row, on the other hand, only increments when a pivot has actually been placed. If the current column holds only zeros in the relevant area, c advances but pivot_row stays put.

Why decouple these two indices

If c and pivot_row advanced together, we'd be assuming there is a pivot in every column — that the matrix is "full" in terms of constraints. Nothing guarantees that. A column may be entirely zero in the region we're examining, either because all constraints concerning that unknown have already been consumed higher up, or because none existed. The column is then called free, and the pivot staircase makes a jump over it. Decoupling the two indices is exactly what lets us handle that jump cleanly. It's also the mechanism that will later let us count the real pivots — the rank — without being fooled by free columns.

A complete example, from start to finish

Let's go back to the system from the previous section and take it all the way to its RREF. It's small, it's square, it's invertible — the perfect setup for nailing down the mechanics without getting tangled up in edge cases.

$$\begin{aligned} 2x + 3y &= 8 \\ -7x + 4y &= 2 \end{aligned}$$

We write the augmented matrix: the coefficients of the unknowns to the left of the bar, the right-hand sides to the right.

L_1 : [ 2 3 | 8 ] L_2 : [ −7 4 | 2 ]

Column 1, looking for a pivot. Row $L_1$ shows a $2$: that'll do. No need to swap. Normalization: divide the whole row $L_1$ by $2$ so its pivot becomes $1$.

L_1 : [ 1 1.5 | 4 ] L_2 : [ −7 4 | 2 ]

Next, the elimination. Row $L_2$ contains a $-7$ in the pivot column — that's what we want to kill. Since the pivot equals $1$, we get $k = -7$ directly, so $L_2 \leftarrow L_2 - (-7) \cdot L_1$, i.e., add $7 \cdot L_1$ to $L_2$. Column 1: $-7 + 7 \times 1 = 0$. Column 2: $4 + 7 \times 1.5 = 14.5$. Augmented column: $2 + 7 \times 4 = 30$.

L_1 : [ 1 1.5 | 4 ] L_2 : [ 0 14.5 | 30 ]

First pivot placed. We increment pivot_row and move on to the next column.

Column 2, looking for a pivot. Row $L_2$ contains $14.5$ in this column, nonzero, that's a pivot. No swap. Normalization: divide the whole row by $14.5$. The right-hand side becomes $30 / 14.5 = 60/29$, about $2.069$.

L_1 : [ 1 1.5 | 4 ] L_2 : [ 0 1 | 60/29 ]

Elimination. In RREF we also kill the entry above the pivot: the $1.5$ in $L_1$. With $k = 1.5$, we set $L_1 \leftarrow L_1 - 1.5 \cdot L_2$. Column 1: unchanged (since $L_2$ is zero there). Column 2: $1.5 - 1.5 \times 1 = 0$. Augmented column: $4 - 1.5 \times 60/29 = 116/29 - 90/29 = 26/29$, about $0.897$.

L_1 : [ 1 0 | 26/29 ] L_2 : [ 0 1 | 60/29 ]

There's the RREF. No more pivots to place, the algorithm halts. The reading is immediate: $x = 26/29$ and $y = 60/29$. No inverse, no back-substitution into another equation. The discipline of the three elementary operations, applied in the right order, surfaced the solution on its own. Sanity check: $2 \times 26/29 + 3 \times 60/29 = 232/29 = 8$, and $-7 \times 26/29 + 4 \times 60/29 = 58/29 = 2$. The machine doesn't lie.

When the staircase skips a step: free columns

Things don't always go this smoothly. Sometimes, when you reach a column, you notice it contains only zeros in the region you're still allowed to search (below the most recently placed pivot row). No candidate. The algorithm advances without incrementing pivot_row: the staircase of pivots has skipped a step.

A column without a pivot is called a free column, and the unknown that goes with it is a free variable. Concretely, this variable isn't pinned down by any direct constraint: you can give it any value, and adjust the other unknowns accordingly. This is exactly "case 3" from the previous section, the one with infinitely many solutions.

The entries of a free column, after reduction, aren't noise: they express the dependence of the pivot unknowns on the free variable. If a particular pivot's row shows $0.625$ in a free column, that means the pivot unknown equals "the right-hand side minus $0.625$ times the free variable." We'll come back to this on the rank page, because it's by counting columns with a pivot and those without that we measure the system's freedom.

Key takeaways

  1. The target is the RREF. Pivots equal to $1$, descending staircase, zeros above and below each pivot, zero rows at the bottom. It's the cleanest form you can reach with the three elementary row operations.
  2. Three operations, and only three. Swap two rows, multiply a row by a nonzero scalar, add a multiple of one row to another. All three are reversible, so the solution set is preserved.
  3. The key formula for elimination. The coefficient $k$ that cancels a term using a pivot equals "what you want to kill, divided by the pivot." When the pivot has been normalized to $1$, the division vanishes and $k$ is simply the value to be cancelled.
  4. Two indices, two roles. The current column advances every round, but the pivot row only increments if a pivot has actually been placed. This decoupling handles free columns and lets the staircase skip steps without breaking.
  5. One engine, multiple uses. This same reduction procedure will compute the determinant, the inverse, and the rank of a matrix in the sections that follow. Understanding Gaussian elimination once sets the stage for the rest of the chapter.
