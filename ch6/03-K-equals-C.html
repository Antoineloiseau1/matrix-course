Chapter 6 · §3

K = C in our structures

Since exercise 00, we've been careful to keep a generic parameter $K$ in every signature of Vector<K> and Matrix<K>. The assignment promised that the bonus from exercise 15 — redoing every exercise with $K = \mathbb{C}$ — "shouldn't require much fixing" if we did our work properly. This chapter cashes in on that promise by walking through, one by one, every operation we coded, to identify what stays the same and what changes subtly when $K$ becomes $\mathbb{C}$.

The most surprising observation is that we won't change a single line of vector.rs or matrix.rs. Just writing a new Complex type that implements the Field trait is enough to make Vector<Complex> and Matrix<Complex> compile and work. That's the power of Rust's generics combined with the abstraction discipline we kept up relentlessly throughout the module.

The mechanism: what really changes

When we instantiate Vector<Complex>, several things happen in cascade. The struct, which was 

pub struct Vector<K> { data: Vec<K> }

becomes, after monomorphization, the equivalent of 

pub struct VectorComplex { data: Vec<Complex> }

that is, an array of Complex instead of an array of f32. Every method — add, dot, norm, etc. — that used to call operations on K now calls those operations' implementations for Complex: Add for Complex instead of Add for f32, mul_add for Complex instead of the one for f32, and so on.

The K: Field bound we placed on every method of Vector and Matrix plays two roles:

  1. At compile time, it guarantees that every algebraic operation used inside the method bodies is defined for the concrete type we chose. If Complex didn't implement Add, compilation would fail with a readable error message.
  2. At run time, it resolves each operation call to the concrete type's specific implementation — at no extra cost (Rust generics are monomorphized, unlike Java generics).

Let's go through every operation in the module to see, concretely, what the substitution $K = \mathbb{C}$ produces.

What stays the same

Addition, subtraction, scaling (ex00)

Vector::add, Vector::sub, Vector::scl, and their matrix counterparts, iterate component by component. For two complex numbers $z = a + bi$ and $w = c + di$, we have $z + w = (a+c) + (b+d)i$ — that's vector addition in the complex plane, exactly what we saw in §2. The code is identical: 

for k in 0..self.size() { self.data[k] += rhs.data[k]; }

For scaling by a complex number $\lambda$, each component is multiplied by $\lambda$; and we saw that multiplying by $\lambda$ corresponds geometrically to a rotation-and-scaling around the origin. Multiplying a complex vector by $i$, for instance, rotates each component by a counterclockwise quarter-turn. That's what the demo output shows: [1, i] × i = [i, -1].

Linear combination (ex01)

linear_combination(vectors, coefs) is nothing more than a sequence of scalings and additions accumulated via fma (fused multiply-add). Every ingredient is available for $\mathbb{C}$, so the operation works out of the box. What's new is that the coefficients can now be complex: we can form i * e_1 + (2 + 3i) * e_2 and obtain a coherent complex vector.

Cross product (ex06)

The cross product, defined in three dimensions, works unchanged on complex coordinates. Its formula, which combines multiplications and subtractions of coordinates, remains valid. Geometrically it's less intuitive (we leave the comfortable terrain of "two real vectors in $\mathbb{R}^3$"), but algebraically there's no obstacle.

Matrix multiplication (ex07)

Matrix::mul_vec and Matrix::mul_mat are accumulations of dot products. Each coefficient of the result is a sum of products of coefficients. Since $\mathbb{C}$ supports both operations, it all goes through. The demo output confirms it: multiplying the complex matrix $[[1+i, 2], [i, 1-i]]$ by the vector $[1, i]$ produces a perfectly coherent complex vector.

Trace, transpose (ex08, ex09)

The trace sums the diagonal coefficients. If the matrix is complex, the trace is complex. No surprises.

The transpose, however, deserves a moment of attention. The exercise 09 brief mentioned, in section XIII.1, the existence of the conjugate transpose (also called the Hermitian transpose), which combines transposition with component-wise conjugation. Our implementation, Matrix::transpose, only performs the geometric transposition: it swaps rows and columns, but does not conjugate the coefficients. That's the simple choice, which is rigorously correct as an algebraic operation over any field, and it's what the assignment asks for. We leave the conjugate transpose as an optional extension (see §4 for the details).

Gauss, row-echelon (ex10)

The Gaussian elimination algorithm is purely algebraic: it combines pivot divisions, subtractions of row multiples, and permutations. None of these operations requires the coefficients to be real. Matrix::row_echelon on a complex matrix produces a complex RREF, exactly the same way it does on a real one. The demo verified it: the RREF of $[[1, 2+i], [3, 4+2i]]$ is the $2 \times 2$ identity matrix, because the two rows were linearly independent as vectors in $\mathbb{C}^2$.

Inverse, rank (ex12, ex13)

Built on Gauss-Jordan, the inverse and rank automatically inherit compatibility with $\mathbb{C}$. The demo inverted the complex matrix $[[1, i], [0, 1]]$ and got $[[1, -i], [0, 1]]$, which you can check by hand: $[[1, i], [0, 1]] \times [[1, -i], [0, 1]] = [[1, -i + i], [0, 1]] = [[1, 0], [0, 1]]$. Correct.

What changes subtly

Determinant (ex11)

The determinant of a complex matrix is itself a complex number. Geometrically, in the real case, the determinant encodes the signed area (or volume) the transformation produces from a unit cube. In the complex case, this simple geometric interpretation goes away: you'd have to think in terms of volumes in $\mathbb{R}^{2n}$ (each complex coefficient has 2 real components), which is much more subtle. For our implementation, nothing needs to change: the recursive formula (cofactor expansion) or the product-of-pivots approach works in any field. The result is just a complex number instead of a real one.

Our demo computed $\det([[1, 2], [i, 1]]) = 1 - 2i$: that's $1 \times 1 - 2 \times i = 1 - 2i$, matching the formula.

Dot product (ex03)

Here's the most interesting subtlety. Our dot product is defined bilinearly:

$$u \cdot v = \sum_{k} u_k \times v_k$$

On reals, $u \cdot u = \sum u_k^2$ is always non-negative, which is exactly the "squared length" of the vector $u$. On complex numbers, however, $u \cdot u$ can be complex and non-positive. For example, with $u = [i, 1]$, we get $u \cdot u = i^2 + 1^2 = -1 + 1 = 0$, even though the vector $[i, 1]$ is obviously not the zero vector.

This pathology is well known. The standard mathematical fix is the Hermitian inner product:

$$\langle u, v \rangle = \sum_{k} \overline{u_k} \times v_k$$

Conjugating the first factor guarantees that $\langle u, u \rangle = \sum |u_k|^2$ is always a non-negative real, and so really does encode a "squared length". The assignment (section VII.1) mentions the Hermitian inner product as an interesting object, but doesn't require it in the implementation. Our dot therefore stays bilinear, in line with the assignment.

The bilinear complex dot product is mathematically "ill-posed" in the sense that it doesn't give a true distance. The assignment accepts that; mathematical rigour would call for the Hermitian inner product. It's documented as an aside, not implemented.

Norm (ex04)

Here's where the Field trait's design shines. Our Euclidean norm is defined by:

$$\|u\|_2 = \sqrt{\sum_{k} |u_k|^2}$$

The 1-norm and infinity norm have analogous definitions based on $|u_k|$. The key point is that all these formulas use the magnitude $|u_k|$, not $u_k$ directly. And $|u_k|$ is a non-negative real number regardless of what $u_k$ is: for a real, it's the absolute value; for a complex number, it's the modulus $\sqrt{a^2 + b^2}$.

That non-negative real number is exactly what the abs(self) -> f32 method of the Field trait returns. The signature, decided back in exercise 00, anticipated this moment: abs doesn't return Self but f32, because the magnitude of an object is always a real number whatever the object is. As a result: Vector<Complex>::norm compiles unchanged and returns an f32, as the assignment requires ("Norms always return real numbers, even for complex-valued vectors").

The demo verified it: the norm of the vector $[3+4i, 0]$ is $\sqrt{|3+4i|^2 + 0} = \sqrt{25} = 5$. A real number.

Angle_cos (ex05) and lerp (ex02)

Two operations stay on the sidelines:

  • angle_cos computes $u \cdot v / (\|u\| \|v\|)$ and returns an f32. On real vectors, that's the cosine of the angle. On complex vectors, it's the real part of the complex dot product divided by the product of the moduli, which doesn't have the same geometric interpretation. The method compiles (thanks to to_f32, which extracts $\Re(z)$) but the result is only partially meaningful. Marginal use case.
  • lerp explicitly requires a Mul<f32, Output = V> bound, that is, the ability to scale the type by a real number. For Vector<Complex>, the generic Mul<K> implementation yields Mul<Complex>, not Mul<f32>. So lerp doesn't apply directly to a complex vector; you'd have to either change the bound or implement Mul<f32> for Complex. Marginal, we leave it.

3D projection (ex14)

The projection function is intrinsically f32-only, by its geometric nature: a 3D projection matrix is a transformation between real spaces. It sits outside the ex15 bonus and isn't meant to be instantiated with complex numbers.

The Field trait, in retrospect

To wrap up, let's take stock of the Field trait and its role as the interface between abstract algebra and our Rust code. Here's the full definition: 

pub trait Field:
    Add<Output = Self> + AddAssign +
    Sub<Output = Self> + SubAssign +
    Mul<Output = Self> + MulAssign +
    Div<Output = Self> + DivAssign +
    Neg<Output = Self> + Copy + PartialEq
{
    fn zero() -> Self;
    fn one()  -> Self;
    fn mul_add(self, a: Self, b: Self) -> Self { self * a + b }
    fn abs(self) -> f32;
    fn to_f32(self) -> f32;
}

Every line tells a story. The bounds (Add, Sub, etc.) capture the algebraic axioms of a field: you can add, multiply, and divide, and these operations return objects of the same type. Copy requires the value to be cheaply duplicable — natural for "lightweight scalars" like f32 or our Complex. PartialEq lets us test whether a value is zero (useful in Gauss to detect columns without a pivot).

There are four methods to provide. zero() and one() are the additive and multiplicative identities, indispensable for initializing matrices. mul_add is the fused multiply-add operation whose importance for numerical precision the assignment underscores. abs and to_f32 are the two "to the reals" methods that anticipate ex15 precisely: abs gives the magnitude (always real), to_f32 gives a projection to the reals (for the ratios inside angle_cos).

This interface was designed, from the start, to make the move to $\mathbb{C}$ trivial. And it kept its promise: we added a single src/complex.rs file, about 130 lines including all the operators, the Display trait, and the impl Field. That's the entire technical effort. No changes anywhere else.
